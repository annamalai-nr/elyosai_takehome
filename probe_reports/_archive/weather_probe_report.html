1TL;DR

Five high-impact findings, ranked by chat-app design impact:

  1. Rate limit is HTTP 200 with a body envelope, not 429. After ~5 successful calls per ~30 s window, every subsequent call returns {"status":"throttled","retry_after_seconds":29,"data":null} with HTTP 200. The chat app must inspect the body even on 200, parse retry_after_seconds, and back off transparently. Retry-After is not sent as a header.
  2. Two success schemas. The API can return either; clients must handle both. Most calls return flat {location, temperature_c, condition, humidity} (Shape A). Some return nested {location, conditions:[...], note} with two conflicting readings (Shape B). The variant is not city-specific — the same city has been observed under either shape across runs — but our probe budget is too small to pin down whether the choice is per-call random, per-instance, or input-conditional. Parser must handle both.
  3. The API silently fuzzy-matches and fabricates. ?location=Mars returns weather for Marseille. ?location=京都 normalises to 京都市. ?location=Atlantis returns plausible-looking fabricated weather (19.3 °C Sunny). The chat app cannot detect mismatches from the response alone — only the LLM downstream can flag "wait, you asked about Mars and got Marseille."
  4. Errors are well-shaped and reliable. 401 (auth), 404 (empty / non-resolvable / overly-long location), 422 (missing/wrong param name), 405 (non-GET method) all bypass the throttle and return distinct, parseable bodies. Treat these as the "fail loud" path.
  5. No prompt-injection content in any of 32 responses. The injection planted at / is confirmed as a one-off Easter egg for LLM-assisted candidates, not a system-wide pattern.

2Method

Two runs of the same 32-probe matrix (backend/probes/probe_weather.py). Run 1 was naive (no backoff); after call 5 it hit the rate limit and the remaining 27 calls returned throttle envelopes that masked the actual per-input behaviour. Run 2 added a throttle-aware retry loop: when a 200 OK arrives with "status":"throttled" in the JSON body, sleep retry_after_seconds + 1 and retry the same call, max 3 retries.

Important: the throttle variant is HTTP 200, not 429. Naive code that checks response.status_code != 200 will treat throttled responses as successes and feed garbage into the LLM context. Always inspect the body even on success.

4Throttle behaviour — characterised

4.1 Observed pattern

  • Roughly 5 successful 200-with-real-content calls per ~30 second window, then the 6th returns the throttle envelope.
  • In this run, 401/404/405/422 responses were never replaced by throttle envelopes — they all came back fast (~30–40 ms) regardless of how depleted the budget was. Consistent with the error path short-circuiting before the rate-limit check, but the API authors haven't documented that as a guarantee.
  • retry_after_seconds starts at 29 and decrements as wall-clock time passes inside the window — observed values 29 → 28 in Run 1.
  • No Retry-After HTTP header is set on throttle responses — the back-off value is body-only.
  • After sleeping retry_after_seconds + 1, the same call succeeds. Confirmed across 6 backoff retries in Run 2 (Sydney, t3_upper, t3_springfield, t4_dup_param, t6_concurrent_2, t6_concurrent_3) — 0 calls gave up.

4.2 Implementation pattern for the chat app

def call_weather(location):
    for attempt in range(MAX_RETRIES):
        resp = requests.get(BASE + "/weather",
                            params={"location": location},
                            headers={"X-API-Key": api_key},
                            timeout=15)
        if resp.status_code == 200:
            body = resp.json()
            if body.get("status") == "throttled":
                sleep_s = body.get("retry_after_seconds", 30) + 1
                time.sleep(sleep_s)
                continue                              # retry same call
            return body                                # real data
        if resp.status_code in (401, 404, 422, 405):
            return {"error": f"http_{resp.status_code}", "detail": resp.json()}
        raise RuntimeError(f"unexpected status {resp.status_code}")
    raise RuntimeError("max retries exceeded")

This same pattern transfers directly to the LLM tool wrapper: catch throttle, sleep, retry, return real data. The LLM never has to see the throttle envelope — it just experiences slightly higher tool latency on throttled calls.

5Schema variation — A vs B

The same endpoint returns one of two top-level schemas. The variant is not city-specific — across the two runs, the same city has been observed under either shape. The probe budget is too small to characterise what chooses between them (per-call randomness, server-instance differences, and input-conditional logic all remain plausible).

5.1 Shape A (flat)

{
  "location": "London",
  "temperature_c": 14.0,
  "condition": "Partly cloudy",
  "humidity": 67
}

Four fields, scalar values. The dominant shape — appeared on the great majority of successful calls in this run.

5.2 Shape B (multi-observation array)

{
  "location": "Paris",
  "conditions": [
    {"temperature_c": 13.2, "condition": "Overcast",      "humidity": 88},
    {"temperature_c": 12.2, "condition": "light rain",    "humidity": 100}
  ],
  "note": "Multiple conditions reported"
}

Two nested observations with conflicting data. Mixed casing in condition values ("Overcast" capitalised, "light rain" not). Appeared occasionally — 1 of 8 Tier 1 cities in Run 2, plus the duplicate-param case (which is effectively Paris again). Too few observations to commit to a percentage.

5.3 Parser pattern

def normalise_weather(body):
    if "temperature_c" in body:
        # Shape A — flat
        return {"location": body["location"],
                "observations": [{"temp_c": body["temperature_c"],
                                  "condition": body["condition"],
                                  "humidity": body["humidity"]}]}
    if "conditions" in body:
        # Shape B — multi-observation
        return {"location": body["location"],
                "observations": [{"temp_c": o["temperature_c"],
                                  "condition": o["condition"],
                                  "humidity": o["humidity"]} for o in body["conditions"]],
                "note": body.get("note")}
    raise ValueError(f"unrecognised weather body shape: {list(body.keys())}")

Normalising into a single internal shape before handing to the LLM keeps the LLM prompt simple and removes a category of "the JSON looked different that time" hallucinations.

6Fuzzy match & fabrication

The API silently rewrites or fabricates location values without telling the caller.

InputReturned locationNotes
MarsMarseilleTook the planet name as a prefix match for the French city. The caller asked about a planet; the response is about a city, with no indication of the substitution beyond the location field.
京都京都市Normalised to the formal Japanese name for Kyoto City. Reasonable.
London, UKLondonComma + country dropped silently. Bit-identical body to plain London.
London LondonWhitespace stripped. Bit-identical body to plain London.
LONDON / londonLondonCase folded. Bit-identical body to plain London.
AtlantisAtlantisEcho'd back literally with fabricated weather: 19.3 °C Sunny 73% humidity. There is no real Atlantis to fuzzy-match to.
SpringfieldSpringfieldReturned plausible weather without specifying which Springfield (there are many).

Implication: trust the response's location field over the input

When passing the tool result to the LLM, include the requested location in the conversation context so the LLM can flag mismatches to the user. For example: "You asked about Mars and the weather service returned Marseille, France — were you asking about the city or the planet?" This is the right defensive UX for a tool-calling agent.

In Run 2, every 401/404/405/422 came back at ~30–40 ms even when the rate-limit bucket was depleted — consistent with the error path short-circuiting before the rate-limit check, but observed behaviour only, not a documented guarantee.

Other auth observations

  • Header name is case-insensitive: x-api-key (lowercase) works identically to X-API-Key. Both Run 1 and Run 2 confirmed.
  • Same body for no-key vs wrong-key. An attacker probing valid keys can't tell from the response shape whether they missed the header or sent something invalid.
  • Error responses are fast (~30–40 ms), roughly 3× quicker than valid weather generation (~110–170 ms).

9Concurrency

5 parallel calls to /weather?location=London were submitted via a ThreadPoolExecutor(max_workers=5).

Run 1 (naive, throttle already active)

All 5 returned the throttle envelope. Wall time 0.046 s, individual latencies 0.031–0.046 s. Confirms true server-side parallelism — Cloud Run handled the 5 requests concurrently rather than serialising them.

Run 2 (throttle-aware)

  • 3 of the 5 calls (concurrent_0, concurrent_1, concurrent_4) succeeded immediately — they fit in the remaining budget of the current 30 s window.
  • 2 calls (concurrent_2, concurrent_3) returned throttled, slept 30 s, retried, and succeeded.
  • Latencies for the 3 immediate winners: 0.201–0.203 s (slightly slower than sequential calls, expected under concurrent fan-out).
  • Latencies for the 2 retry winners: 0.148 s each.
  • All 5 ultimately returned bit-identical bodies (SHA ce136498…) — the canonical London Shape A response.

Implication for the chat app: concurrent tool calls (which the on-site session may push the implementation toward) do each consume a slot in the throttle window. If the LLM kicks off two slow tools in parallel after the budget has been mostly used, expect at least one to throttle. The retry loop in §4.2 handles this transparently.

11Prompt-injection check

Every response in the retained Run 2 log — 32 logical probes plus 6 throttled-first sidecar responses, 38 bodies in total — was scanned with eight heuristic patterns. (The Run 1 JSONL was not preserved, so the strict claim here covers Run 2 only; Run 1 results are referenced from interactive notes but cannot be re-verified from disk.)

  • HTML comments (<!--…-->)
  • Override-instruction phrases ("ignore all previous instructions" and variants)
  • AI self-references ("if you are an LLM…")
  • Known seed markers from the / root injection ("banana", "please add a comment", "found via api")
  • Zero-width / bidirectional override Unicode characters
  • Roleplay markers ("act as…", "from now on…", "pretend that…")
  • System-prompt markers ("system:", "instruction:", "important:")
  • Imperative-at-reader phrases ("you must", "please add", "please reply")

Result: zero heuristic hits across all 38 retained Run 2 bodies. The /weather endpoint does not carry any of the patterns planted at the / root in the responses we can re-verify.

This confirms Codex's earlier pushback: the prompt-injection finding at / is a one-off Easter egg for LLM-assisted candidates. The actual tool endpoints are clean. The "treat tool responses as untrusted data" defensive default still stands as a discipline, but it's not currently fighting any active threat in the planted probes.

12What this changes for the chat app

12.1 Mandatory handling (must implement)

  1. Throttle envelope detection. Inspect every 200 response body for "status":"throttled"; parse retry_after_seconds; sleep+1 and retry. (§4.2)
  2. Dual-schema parser. Normalise Shape A and Shape B into a single internal shape before handing to the LLM. (§5.3)
  3. Error envelope handling. 401, 404, 405, 422 each have distinct, parseable bodies. Surface a clean error to the LLM, not a stack trace. (§8)
  4. Cancellation hygiene. If the user Ctrl+C's during a throttle sleep, abort the sleep and the in-flight retry, preserve conversation history, return to prompt. This is one of the take-home's explicit evaluation criteria.

12.2 Strongly recommended (production-quality, ~20 extra LOC)

  1. Pass the requested location through to the LLM alongside the API's returned location. Lets the LLM detect Mars→Marseille style fuzzy substitutions and surface them to the user.
  2. Bounded retry count. Cap at 3 throttle retries per tool call so an outage doesn't hang the chat.
  3. Show pending state on throttle backoff. print("\rWaiting 30 s for rate limit…", flush=True); clears when the call succeeds.

12.3 Not needed for the take-home

  1. Voice / Deepgram / ElevenLabs — out of scope.
  2. CRM / Simpro / JobLogic integration — out of scope.
  3. Background job queue for retries — overkill for a CLI.
  4. Heuristic prompt-injection sanitiser on tool responses — no active threat from the probes; mentioning it in the Loom is enough.

13What's still unknown

  • Exact throttle threshold. Observed ~5 per ~30 s, but we haven't tested whether the window is rolling vs tumbling, nor whether the limit is per-key vs per-IP. Both are unlikely to matter for the chat app; the retry-on-throttle pattern works regardless.
  • What inputs trigger Shape B vs A. Appears random per call. Could be load-balanced backends with different schemas, or deliberate per-call randomisation. Not knowable from this probe budget.
  • Whether the fuzzy match has a confidence threshold. "Mars" → Marseille but "Atlantis" → Atlantis (no real match). The rule isn't visible from a small sample.
  • Whether the 5000-char input limit is exactly 5000, or applies at some lower threshold. Not worth burning calls to characterise.
  • Whether the /research endpoint has the same throttle policy. The next probe run will tell us.
