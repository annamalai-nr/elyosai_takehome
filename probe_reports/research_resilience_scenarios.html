/research endpoint

Resilience Scenarios

Four scenarios showing how the proactive pacer, bounded concurrency, and retry stack protect the chat app against the /research API's rate-limiting, timeouts, and cancellation behaviour. Each scenario compares a naive approach (no pacer) against the full resilience stack.

API Constraints

Server-side:

  • ~5 requests per ~30 s sliding window, shared across /weather and /research
  • Throttle is HTTP 200 with body {"status":"throttled", "retry_after_seconds":N}, not 429
  • Response time: 3–8 s per successful call (vs ~150 ms for /weather)
  • Server-side timeout at ~15 s returns {} (empty JSON, still HTTP 200)
  • Cancelled calls still consume a throttle slot on the server

Client-side resilience stack:

  • pacing.py — wait_for_budget(): proactive sliding-window tracker with shared "api" key
  • agent.py — asyncio.Semaphore(1) for /research (serialises concurrent calls)
  • elyos_client.py — throttle retry (up to 2 retries, sleeps retry_after + 1 s)
  • elyos_client.py — client-side timeout at 20 s with 1 timeout retry
  • parsers/research.py — empty-body detection ({} → kind="timeout")
  • cli_chat.py — SIGINT cancellation with user warning

How the Proactive Pacer Works

The server enforces a rate limit of ~5 requests per ~30 s sliding window, but it does not use HTTP 429. Instead it returns HTTP 200 with a JSON body: {"status":"throttled"}. A naive client would not know it was throttled until it parsed the response body — and by then it has already wasted a round trip.

The proactive pacer (wait_for_budget in pacing.py) mirrors the server's sliding window on the client side so that calls are delayed before they hit the server, not after. It works like this:

Client-side budget: a deque of timestamps

The pacer maintains a single shared deque (named "api") that records the timestamp of every API call — weather or research — within the current window. Before each call, wait_for_budget runs this logic:

  1. Expire old entries. Remove any timestamps older than window_s (30 s) from the front of the deque.
  2. Check capacity. If the deque has fewer than max_requests_per_window (5) entries, append the current timestamp and proceed — the call is allowed.
  3. Wait if full. If the deque is at capacity, calculate how long until the oldest entry expires (window_s − age_of_oldest), print “Rate budget full, pacing for Ns…”, sleep that duration, and re-check.

The timestamp is recorded before the HTTP call begins, not after it completes. This is critical: if the call is cancelled mid-flight or times out, the slot is already accounted for in the client-side budget — matching the server's behaviour where cancelled calls still consume a slot.

A single asyncio.Lock protects the deque so concurrent calls cannot double-book the same slot. Combined with the per-endpoint semaphore (Semaphore(1) for /research), calls are serialised and budget-gated before they ever reach the network. 

Budget deque (capacity = 5, window = 30s):

  [t=0.0, t=0.1, t=5.2, t=11.0, t=17.5]   ← 5/5 full
   ↑                                          ↑
   oldest (expires at t=30.0)                 newest

  At t=24.0, a new call arrives:
    • Expire: nothing (oldest is 24s old, < 30s window)
    • Check:  5/5 → full
    • Wait:   30.0 − 24.0 = 6.0s
    • Print:  "Rate budget full, pacing for 6s..."
    • Sleep 6s, re-check, oldest expired → 4/5 → proceed

Scenario 1: Throttle Storm

The LLM issues 3 parallel research_topic calls in a single turn. Two slots are already consumed from earlier weather lookups in the same window, leaving 3 slots remaining.

User prompt: “Compare solar energy, wind energy, and nuclear power”

LLM responds with 3 tool calls: research_topic("solar energy"), research_topic("wind energy"), research_topic("nuclear power").

Budget state: 2 of 5 slots already used (from prior weather calls). 3 slots remain.

Case 1 — No Pacer

All 3 calls fire simultaneously. No client-side budget tracking.

  • 0.0scall_1 ("solar energy") fires
  • 0.0scall_2 ("wind energy") fires
  • 0.0scall_3 ("nuclear power") fires
  • 0.04scall_1 accepted (slot 3 of 5)
  • 0.04scall_2 accepted (slot 4 of 5)
  • 0.04scall_3 accepted (slot 5 of 5)
  • 5.2scall_1 returns data ✓
  • 6.8scall_2 returns data ✓
  • 7.5scall_3 returns data ✓
~7.5 s wall time. 3 API calls. All 3 succeed — but the window is now at 5/5. The next call in the same window (weather or research) will be throttled. No budget tracking means the client has no idea.

Case 2 — With Pacer

Semaphore(1) serialises calls. wait_for_budget gates each one.

  • 0.0ssem acquired → budget 3/5 → call_1 fires
  • 0.0scall_2, call_3 wait for semaphore
  • 5.2scall_1 returns data ✓ — sem released
  • 5.2sbudget 4/5 → call_2 fires
  • 11.0scall_2 returns data ✓ — sem released
  • 11.0sbudget 5/5 → call_3 fires
  • 17.5scall_3 returns data ✓
~17.5 s wall time (slower). 3 API calls, 0 throttles. Budget is at 5/5 but the client knows it — the next call will wait for the window to slide instead of hitting a surprise throttle.
Trade-off: with only 3 slots remaining, both cases succeed. The pacer is slower (serial vs parallel) but leaves the client with an accurate budget. The real difference emerges when the window is tighter — see Scenario 2.

Scenario 2: Burst Into a Nearly Full Window

The LLM issues 3 parallel research_topic calls, but 4 slots are already consumed from earlier weather lookups. Only 1 slot remains in the window.

User prompt: “Compare solar energy, wind energy, and nuclear power”

LLM responds with 3 tool calls. Budget state: 4 of 5 slots used. 1 slot remains. Prior 4 calls were made 5–10 s ago (they expire in 20–25 s).

Case 1 — No Pacer

All 3 fire simultaneously. Only 1 gets a slot; 2 are throttled.

  • 0.0scall_1, call_2, call_3 all fire simultaneously
  • 0.04scall_1 accepted (slot 5 of 5)
  • 0.05scall_2 throttled — retry_after: 25
  • 0.05scall_3 throttled — retry_after: 25
  • 6scall_1 returns data ✓
  • 6sUser sees call_1 result but is still waiting for calls 2 and 3…
  • 26scall_2 retries → window slid, prior calls expired → accepted
  • 26scall_3 retries → accepted
  • 32scall_2 returns data ✓
  • 33scall_3 returns data ✓
~33 s wall time. 5 API requests (2 wasted on throttle responses). All 3 eventually succeed, but the user waited ~26 s for a retry cycle, seeing “Rate-limited, retrying in 26s…” with no advance warning.

Case 2 — With Pacer

Semaphore(1) serialises calls. wait_for_budget gates each one.

  • 0.0ssem acquired → budget 5/5 → call_1 fires
  • 0.0scalls 2, 3 wait for semaphore
  • 6scall_1 returns data ✓ — sem released
  • 6sbudget still 5/5 (prior entries not yet expired) — must wait
  • 6sCLI: “Rate budget full, pacing for 14s…”
  • 20soldest entry expires → budget 4/5 → call_2 fires
  • 26scall_2 returns data ✓ — sem released
  • 26smore entries expired → budget 3/5 → call_3 fires
  • 32scall_3 returns data ✓
~32 s wall time (comparable). But only 3 API requests — zero wasted on throttle. The user sees “pacing for 14s” up front instead of a surprise “Rate-limited, retrying” midway through.
Key insight: the total wait is similar in both cases because the underlying constraint is the same (the window must slide before more calls can go through). The difference is when the user learns about the wait: the pacer tells you up front (“pacing for 14s”), while the no-pacer path wastes 2 API requests discovering the limit reactively. With higher call counts, the reactive retries also compete with each other for slots, amplifying the delay.

Scenario 3: Mid-Flight Cancellation

The user presses Ctrl+C during a research call (common because /research takes 3–8 s). The cancelled call still consumed a server throttle slot. The question is: does the client know about that consumed slot when the next call happens?

User asks “Research quantum computing”, then presses Ctrl+C at 2 s, then asks a new question

Budget state: 4 of 5 slots already used. 1 slot remains.

Case 1 — No Pacer

Client has no budget tracking. Cancelled call's slot is invisible.

  • 0.0sresearch("quantum computing") fires (server slot 5 of 5)
  • 2.0sUser presses Ctrl+C → CancelledError
  • 2.0sCLI prints cancellation message
  • 2.1sUser asks: “Research solar energy instead”
  • 2.1sresearch("solar energy") fires immediately
  • 2.15sServer returns {status:"throttled", retry_after:27}
  • 2.15sClient sleeps 28 s (retry_after + 1)
  • 30sRetry → succeeds
  • 36sReturns data ✓
~34 s wasted on a surprise throttle. The user sees “Rate-limited, retrying in 28s…” and thinks: “Why? I only made one call!” The cancelled call's slot was invisible to the client.

Case 2 — With Pacer

wait_for_budget records the slot before the HTTP call begins.

  • 0.0swait_for_budget records slot (budget: 5/5)
  • 0.0sresearch("quantum computing") fires
  • 2.0sUser presses Ctrl+C → CancelledError
  • 2.0sCLI: “Cancelled. The interrupted API call may still count against the rate limit.”
  • 2.1sUser asks: “Research solar energy instead”
  • 2.1swait_for_budget: 5/5 used → must wait ~28 s
  • 2.1sCLI: “Rate budget full, pacing for 28s…”
  • 30sWindow slides → budget opens → call fires
  • 36sReturns data ✓ (no throttle, no retry)
Same ~34 s total wait — but the wait is proactive (client-side pacing) instead of reactive (server throttle + retry). The user sees “pacing for 28s” immediately, not a surprise throttle. And the call succeeds on the first attempt — no wasted API request.
Why it works: wait_for_budget appends the current timestamp to the deque before the HTTP call begins (pacing.py, line 40). If the call is cancelled mid-flight, the slot is already recorded in the client-side budget, matching the server's behaviour. This is the key design decision that makes cancellation safe.

Scenario 4: Server-Side Timeout

The user asks to research a garbage or pathological input. The server hangs for ~15 s and returns {} (empty JSON, HTTP 200). This is not a client timeout — the server genuinely spent 15 s and returned nothing useful.

User asks “Research a” (single character — triggers server timeout)

Budget state: 0 of 5 used. 5 slots available.

Case 1 — No Pacer, No Parser

Raw HTTP call with no empty-body detection.

  • 0.0sresearch("a") fires
  • 15.0sHTTP 200 returns — body is {}
  • 15.0sEmpty dict passed to LLM as tool result
  • 15.0sLLM receives no data — hallucinates or gives confused answer
15 s of silence, then the LLM gets an empty tool result. It has no signal that a timeout occurred. It may hallucinate research content, tell the user nothing was found, or produce a confused response. The consumed slot is also invisible to the client.

Case 2 — With Full Stack

Pacer tracks the slot. Parser detects empty body. Prompt instructs LLM.

  • 0.0swait_for_budget records slot (budget: 1/5)
  • 0.0sresearch("a") fires
  • 0.0sCLI: “Researching a… (Ctrl+C to cancel)”
  • 15.0sHTTP 200 returns — body is {}
  • 15.0sparsers/research.py: if not data → ResearchResult(kind="timeout")
  • 15.0sEnveloped as {"untrusted": true, "data": {"kind":"timeout", "message":"..."}}
  • 15.0sLLM follows prompt rule: “tell the user and suggest retrying”
Same 15 s wait (unavoidable — it's a server-side timeout), but the outcome is clean: the parser converts {} into a structured kind="timeout" signal, the prompt instructs the LLM to explain it, and the pacer correctly tracked the consumed slot. User sees: “The research timed out. Try a more specific topic or try again.”
Note: the pacer cannot prevent server-side timeouts — that is a server behaviour, not a rate-limit issue. But the pacer still matters here: the 15 s call consumed a slot, and the pacer tracked it. Without it, the client would not know that 1 of 5 slots is now gone for the remainder of the window. The parser and prompt layers handle the user-facing recovery.
