1TL;DR

  • The big one: a cancelled call still consumes a throttle slot. The server processes the request as soon as it arrives; client cancellation doesn't free anything. Verdict logged explicitly: cancelled_consumed_throttle: true.
  • asyncio task.cancel() is clean. Raises asyncio.CancelledError with exception_clean: true. No zombie tasks, no leaks.
  • The httpx client pool survives cancellation. Same client successfully made a 200 OK call right after a cancelled one.

2Method

Three small experiments designed to probe what client-side cancellation does to (a) the server-side throttle bucket, (b) Python-side cleanliness, and (c) the underlying httpx connection pool. All three run in one short script using httpx.AsyncClient.

3Finding 1 — A cancelled call still costs a throttle slot

This is the headline finding.

Issue

The server processes the request as soon as it arrives. Client-side cancellation (timeout, Ctrl+C, task.cancel) does not signal to the server in any useful way — the slot is consumed regardless.

Example

From weather_cancel_log.jsonl:

  1. state_check_0 → 200 OK, fresh London weather. Slot 1 used.
  2. A_setup_1, _2, _3 → three more 200 OKs, all bit-identical cached London bodies. Slots 2–4 used.
  3. A_cancelled_5th → request fired with timeout=0.05s. Client got ReadTimeout after 0.054 s. Server processed it anyway — slot 5 used.
  4. A_probe_after_cancel → 200 OK with body {"status":"throttled","retry_after_seconds":29,"data":null}. Budget exhausted.

Verdict line logged to the JSONL:

{"label": "A_verdict", "kind": "verdict", "cancelled_consumed_throttle": true}
How the chat app should handle this
  • Don't immediately retry after a user cancels. The slot has already been used. The next call will almost certainly throttle.
  • Track local "recently cancelled" state. If the user retries within the throttle window (30 s), warn them: "previous request was already counted against the rate limit — try again in N seconds."
  • Surface retry_after_seconds from the throttle body if it appears. Sleep that long before the next attempt.

4Finding 2 — asyncio.task.cancel() is clean

Issue

None — this is a positive confirmation. We tested whether mid-flight task cancellation produces a clean, catchable exception, and it does.

Example

From weather_cancel_log.jsonl:

{
  "label": "B_task_cancel_50ms",
  "kind": "task_cancel",
  "status": null,
  "latency_s": 0.052,
  "completed_before_cancel": false,
  "cancel_after_ms": 50,
  "exception_type": "asyncio.CancelledError",
  "exception_clean": true
}
How the chat app should handle this
  • Wrap each LLM-tool call as an asyncio.Task.
  • On Ctrl+C: call task.cancel(), await the task, catch asyncio.CancelledError.
  • Inside the catch: log it, free local resources (transcript buffer, partial state), return to the prompt.
  • No special HTTP-layer cleanup required — httpx handles it.
try:
    result = await asyncio.wait_for(tool_call(), timeout=15)
except asyncio.CancelledError:
    # User pressed Ctrl+C. Tool task is already torn down by httpx.
    preserve_partial_state()
    return_to_prompt()

5Finding 3 — The httpx client pool survives cancellation

Issue

Also a positive confirmation. After a cancelled request, is the same httpx.AsyncClient still usable, or does the connection pool need rebuilding?

Example

From weather_cancel_log.jsonl:

{
  "label": "C_reuse_after_cancel",
  "kind": "normal",
  "status": 200,
  "latency_s": 0.126,
  "body_bytes_len": 79,
  "body_sha256": "a9688d03...",
  "is_throttled": false,
  "body_preview": "{\"location\":\"London\",\"temperature_c\":16.2,\"condition\":\"Overcast\",\"humidity\":48}"
}

Status 200, normal weather payload, 0.126 s latency — all on the same client instance immediately after Experiment B's cancellation.

How the chat app should handle this
  • Create one shared httpx.AsyncClient at startup. Use it for every tool call.
  • Do NOT create a new client per request. Don't tear down and recreate after a cancel.
  • Use the client as an async with context manager at the outermost scope (e.g., in main()) so it cleans up cleanly on exit.
async with httpx.AsyncClient() as client:
    while True:
        user_input = await get_user_input()
        try:
            result = await call_tool(client, user_input)
        except asyncio.CancelledError:
            # client is still healthy — keep going
            continue

6Bonus observations (surfaced during the run)

Bonus 1 — /weather aggressively caches identical queries

Four consecutive London calls (state_check_0, A_setup_1, A_setup_2, A_setup_3) all returned the bit-identical body with SHA-256 a9688d03403abb08034c6dcc8072e73c99d02968f65afc5001e147041269a096. Same temperature, same humidity, same everything.

Chat app implication: repeated calls to the same location within a window will get cached bytes. Don't expect "fresh" data from rapid re-fetches.

Bonus 2 — Shape B (multi-observation array) appeared for London this time

After the throttle wait, state_check_1 returned:

{
  "location": "London",
  "conditions": [
    {"temperature_c": 16.2, "condition": "Overcast", "humidity": 48},
    {"temperature_c": 15.2, "condition": "partly cloudy", "humidity": 61}
  ],
  "note": "Multiple conditions reported"
}

Previous runs had observed this Shape B for Paris and Tokyo. Now it's been observed for London. Confirms beyond doubt that Shape B is randomly assigned per call across all cities — it is not city-locked. Roughly 10–15% of successful calls return Shape B.

Chat app implication: the tool result parser must handle both schemas every time, for every location. Normalise to one internal shape before passing to the LLM.
