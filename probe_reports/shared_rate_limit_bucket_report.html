Elyos API Shared Rate-Limit Bucket Probe

Cross-endpoint interleaving probe for /weather and /research. Generated 2026-05-19.

Executive Summary

The /weather and /research endpoints share the same effective rate-limit bucket for the tested API key. This was not proven by the earlier same-endpoint probes; it required interleaving the two endpoints within one throttle window.

The observed bucket is approximately 5 successful content responses per ~30 second window. After those five slots are consumed by any mix of the two endpoints, the next content request returns HTTP 200 with a throttle envelope.

Why This Probe Was Needed

Earlier reports established that both endpoints use the same throttle envelope shape: 

{"status":"throttled","message":"Rate limit exceeded. Please wait.","retry_after_seconds":29,"data":null}

However, the earlier /research report explicitly listed shared bucket behavior as unknown: same-endpoint sequences had been tested, but cross-endpoint interleaving had not. This probe closes that gap.

Method

Both experiments used one API key and waited for a clean window before starting. A preflight request confirmed the API was not currently throttled, then the probe slept 32 seconds to clear the preflight slot before starting the measured sequence.

The probe intentionally avoided uncontrolled load. It sent the minimum number of calls needed to test whether slots consumed by one endpoint affect the other endpoint.

Experiment A: Weather Exhausts Research

Question

If five successful /weather calls consume the window, does the next /research call throttle?

Sequence

clean window
5 successful /weather calls
immediate /research call

Verdict

/research throttled immediately after five successful /weather calls. This supports a shared rate-limit bucket.

Experiment B: Research Consumes The Fifth Shared Slot

Question

If four /weather calls succeed and the fifth slot is consumed by /research, does the next /weather call throttle?

Sequence

clean window
4 successful /weather calls
1 successful /research call
immediate /weather call

Observed Output

CallEndpointResultLatencyNotes
1/weathersuccess0.12 sLondon weather payload
2/weathersuccess0.12 sLondon weather payload
3/weathersuccess0.12 sLondon weather payload
4/weathersuccess0.12 sLondon weather payload
5/researchsuccess3.48 sFresh research payload
6/weatherthrottled0.03 sretry_after_seconds=26

Verdict

The successful /research call consumed the fifth shared slot. The next /weather call returned the throttle envelope. This confirms that both endpoints draw from the same effective bucket.

Chat App Implications

  • Configure both endpoints with the same local rate-limit group, for example rate_limit_group: "elyos_api".
  • Keep endpoint-specific concurrency limits separate: /weather can be more concurrent than /research, but both must spend from the same request budget.
  • A user asking for weather plus research in one turn can exhaust the same shared bucket. The harness should coordinate these calls instead of treating each endpoint as having its own independent five-call window.
  • Continue treating the server's retry_after_seconds as authoritative. The local shared-budget limiter is a conservative preflight guard, not a replacement for reactive throttle handling.

Remaining Caveats

  • This proves shared behavior for the tested API key and current deployment. It does not prove whether the bucket is per key, per IP, global, or some combination.
  • The exact server-side algorithm is still approximate. The observed behavior is consistent with roughly five successful content calls per sliding ~30 second window.
  • The client should still parse every HTTP 200 response body because throttles are returned as body-side envelopes, not HTTP 429 responses.
Source: controlled cross-endpoint interleaving probe run from the repository root on 2026-05-19. The literal API key is intentionally not recorded in this report.
