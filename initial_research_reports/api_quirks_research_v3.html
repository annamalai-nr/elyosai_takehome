4Weather APIs — what to expect

Trimmed to the patterns that map to probes against /weather. Vendor history and billing weirdness removed.

4.1 City-resolution inconsistency

weather Ambiguous city names ("Springfield", "London" — multiple real ones, Ontario vs UK) resolve to different places depending on input format and geolocation hints. Returned coordinates and timezone often don't match the city the user meant. Names with whitespace, hyphens, or odd casing can hit different code paths.

4.2 Field omission vs null

weather Weather APIs often omit fields entirely (no key at all) rather than returning null for missing data. UV index, gusts, snowfall genuinely don't exist at every location. Code that assumes the key is always present crashes; code that checks key existence works.

4.3 Unit drift across queries / endpoints

weather Same provider, different endpoints can return temperatures in different units (Kelvin vs Celsius vs Fahrenheit) depending on the units param default and the endpoint version. Apps that work for current weather but break on forecast often hit this.

4.4 Stale / aggressive caching

weather Weather APIs aggressively cache for cost reasons. Same location queried twice within a few minutes returns identical data — sometimes identical timestamps. Tests that rely on "the response changed" fail.

4.5 Transient 5xx from upstream data partners

weather Weather APIs aggregate from underlying sources (NOAA, ECMWF). When one source is degraded, some locations return 500/502 even while others succeed. Surfaces as flaky errors uncorrelated with input.

4.6 Soft limits below documented numbers

weather Provider-side throttling tends to be harsher than published limits. The soft limit (clustered traffic, sustained bursts) is the one that bites in practice.

5Research / search APIs — what to expect

Trimmed to patterns that map to probes against /research. Credit pricing, lawsuits, and provider-specific freshness/recency weaknesses removed.

5.1 Latency variance with long tail

search Tavily's "advanced search" has published p50 around 1s but real-world tails routinely run 3–8 seconds and occasionally 10s+. Other research APIs follow similar shapes. The take-home's "3–8 seconds" framing matches this directly.

5.2 Content-quality variance on identical topics

search Same topic queried at different times can return very different content — different sources, different snippet quality, different summary depth. The API "succeeds" with 200 OK, but the body has more or less useful material run to run.

5.3 JS-rendered or scraper-blocked pages return empty content

search Research APIs render pages offline. JS-heavy SPAs or pages that block scrapers return empty or partial content. The call succeeds; the result is just thin.

5.4 Hallucination in synthesised summaries

search Perplexity-style APIs that LLM-summarise search results can hallucinate even when the underlying sources are accurate. Confident-sounding wrong content. If /research simulates this, expect plausible-but-fabricated answers on niche or impossible topics.

5.5 Snippet truncation varies

search Snippet length is provider-determined. The same query can return different snippet lengths from the same source URL depending on internal logic. Code that expects "snippet always contains X" breaks.

5.6 Authentication header convention

search Brave Search uses X-Subscription-Token; Tavily uses an api_key body param; Serper uses X-API-KEY header; Perplexity uses Authorization: Bearer. The take-home uses X-API-Key — matches Brave's convention. Worth knowing in case auth quirks surface.

6LLM client layer — quirks in your own code

These live in your code, not in Elyos's mocks. They're what the take-home's "implementation quality" criterion grades. Use the official OpenAI or Anthropic SDK directly (the take-home says so explicitly).

6.1 Streaming tool-call arguments arrive as partial JSON

llm The function arguments field is delivered character-by-character as deltas. Most intermediate states are invalid JSON. Accumulate deltas into a buffer; only parse when the tool-call block is marked done. Don't json.loads() each chunk.

6.2 finish_reason: "stop" on truncated output

llm Truncation should return finish_reason: "length", but in practice "stop" is sometimes returned on actually-truncated responses. Don't trust finish_reason alone — also compare actual length to expected.

6.3 Mid-stream error events lack the choices array

llm When an OpenAI stream errors mid-flight, the error chunk is shaped like {"error": {"message": "...", "code": 500}} — no choices. Clients that assume every chunk has choices crash on deserialization. Handle the error shape explicitly.

6.4 SSE empty events crash naive parsers

llm SSE retry directives and comment lines come through as empty data: events. Skip empties; don't try to parse them.

6.5 SSE idle disconnect on long tool calls

llm If the LLM is "thinking" silently for >30s during a tool loop, intermediate networks may kill the SSE connection. Affects all providers. Use streaming for long requests, but also handle reconnect.

6.6 Anthropic 529 overloaded_error vs 429 rate limit

llm If you choose Anthropic: 429 = your account hit a rate limit (slow down). 529 = provider overload (wait ~30s and retry). Treating them identically wastes rate budget or kills recoverable sessions.

6.7 Cancellation hygiene

llm When the user cancels mid-stream, free the LLM SSE connection, cancel the in-flight tool call task, preserve the partial response in conversation history, and return cleanly to the prompt. Hanging tasks and leaked connections are the deepest tell for "did this engineer actually think about cancellation."

7Cross-cutting patterns

7.1 Idempotency on retries

cross-cut Only retry tools that are idempotent or read-only. /weather and /research appear idempotent, so retries should be safe, but verify.

7.2 Bounded exponential backoff + jitter

cross-cut Exponential (1, 2, 4, 8, 16s) + random jitter ±1s. Without jitter, distributed clients synchronise and cause thundering herds. Bound total retries.

7.3 Honour Retry-After when present

cross-cut Naive backoff that ignores the header retries too early and gets re-rate-limited.

7.4 200 OK with error body

cross-cut Some APIs return 200 with an error field in the response body. Inspect body shape before assuming success.

7.5 Schema drift tolerance

cross-cut Tolerate unknown fields; don't switch-case-exhaustively over enum values.

7.6 Timeout that hangs vs errors

cross-cut Some APIs return a clean error on timeout; others stop sending bytes. Always set explicit client-side read timeouts shorter than your end-user deadline.

7.7 Stale caching on repeated queries

cross-cut Identical queries may return identical responses for a window. Cache-buster params help diagnose.

7.8 Concurrent-request interference

cross-cut Some APIs serialise requests; some have shared per-tenant resources that race. Probe under parallel load.

9Day-1 probe checklist

Run this against /weather and /research before writing handling code. Log only confirmed observations to DISCOVERIES.md with: behaviour observed, reproduction steps, impact, handling decision.

A. Input edge cases

  1. Empty, single character, very long (5,000 chars), Unicode (京都, emoji), whitespace-only
  2. SQL-injection-y inputs ('; DROP TABLE) — should be safe
  3. Unknown locations / topics ("Atlantis", "Mars", "the colour of Tuesday") — does it default, error, or fabricate?
  4. Ambiguous cities ("Springfield", "London")
  5. Odd casing/punctuation ("LONDON", "lon-don")

B. Response shape

  1. Same input twice — same response? Same keys? Same content?
  2. Different inputs — consistent shape across them?
  3. Check for missing keys, null values, unexpected extras
  4. Body ever contain {"error": "..."} with 200 status?
  5. /weather: are units consistent across queries?
  6. /research: does summary structure (length, paragraphs, depth) vary?

C. Latency profile

  1. 20 calls to /weather — distribution, p95, p99
  2. 5–10 calls to /research — distribution, tail behaviour
  3. Idle 60s then call — cold-start spike?

D. Error handling

  1. Burst 50–200 calls — 429? Does it carry Retry-After?
  2. Sustained sampling — 5xx transient rate?
  3. Error body shape — JSON, HTML, text, empty?
  4. Missing / wrong API key behaviour

E. Connection & concurrency

  1. 1-second client timeout — clean error or hang?
  2. Cancel /research mid-flight — does server complete anyway?
  3. Two /research concurrently — independent or serialised?

F. Caching

  1. Same query 5 times — identical body? Identical timestamps?
  2. Cache-buster param — behaviour change?

G. LLM client (your code)

  1. Tool-call argument parser handles partial JSON cleanly
  2. Cancellation preserves partial response, frees resources
  3. Truncation detection beyond finish_reason
  4. Mid-stream error events don't crash client

10Probe script — run before writing the implementation

Drop into probe.py, paste the API key, run once, read output, log findings.

import asyncio, httpx, time
from collections import Counter

BASE = "https://elyos-interview-907656039105.europe-west2.run.app"
HEAD = {"X-API-Key": "X_API_KEY"}

async def probe(client, endpoint, param, value, label):
    t0 = time.monotonic()
    try:
        r = await client.get(f"{BASE}/{endpoint}", params={param: value},
                             headers=HEAD, timeout=15.0)
        dt = time.monotonic() - t0
        try:
            j = r.json()
            keys = sorted(j.keys()) if isinstance(j, dict) else type(j).__name__
        except Exception:
            keys = "NOT_JSON"
        ra = r.headers.get("retry-after", "-")
        print(f"[{label}] {r.status_code} {dt:.2f}s retry-after={ra} keys={keys}")
        print(f"    body: {r.text[:240]!r}")
        return (r.status_code, dt, keys, r.text[:240])
    except Exception as e:
        dt = time.monotonic() - t0
        print(f"[{label}] EXCEPTION {dt:.2f}s {type(e).__name__}: {e}")
        return (None, dt, None, str(e))

async def main():
    async with httpx.AsyncClient() as c:
        # A. input edges
        for ep, p, v, lbl in [
            ("weather", "location", "",         "w/empty"),
            ("weather", "location", "London",   "w/normal"),
            ("weather", "location", "LONDON",   "w/upper"),
            ("weather", "location", "京都",      "w/unicode"),
            ("weather", "location", "Atlantis", "w/fake"),
            ("weather", "location", "Springfield", "w/ambig"),
            ("weather", "location", "x"*5000,   "w/long"),
            ("weather", "location", "'; DROP TABLE--", "w/sqli"),
            ("research", "topic",    "",                                    "r/empty"),
            ("research", "topic",    "solar energy",                        "r/normal"),
            ("research", "topic",    "the colour of the smell of Tuesday",  "r/impossible"),
            ("research", "topic",    "x"*5000,                              "r/long"),
        ]:
            await probe(c, ep, p, v, lbl)

        # C. latency distribution
        print("\n--- /weather x20 ---")
        lats = []
        for _ in range(20):
            res = await probe(c, "weather", "location", "London", "w/lat")
            lats.append(res[1])
        lats = sorted(lats)
        print(f"min={lats[0]:.2f} p50={lats[10]:.2f} p95={lats[19]:.2f}")

        print("\n--- /research x5 sequential ---")
        for i in range(5):
            await probe(c, "research", "topic", f"renewable energy {i}", f"r/lat{i}")

        # D. burst
        print("\n--- /weather burst x50 parallel ---")
        out = await asyncio.gather(*[
            probe(c, "weather", "location", f"city-{i}", f"burst{i}")
            for i in range(50)
        ])
        print(f"codes: {dict(Counter(x[0] for x in out))}")

        # E. concurrent research
        print("\n--- /research concurrent x3 ---")
        t0 = time.monotonic()
        await asyncio.gather(
            probe(c, "research", "topic", "alpha", "r/c1"),
            probe(c, "research", "topic", "beta",  "r/c2"),
            probe(c, "research", "topic", "gamma", "r/c3"),
        )
        print(f"wall: {time.monotonic() - t0:.2f}s")

        # F. caching
        print("\n--- /research cache x3 ---")
        for i in range(3):
            await probe(c, "research", "topic", "renewables", f"r/cache{i}")
        print("\n--- /weather cache x5 ---")
        for i in range(5):
            await probe(c, "weather", "location", "London", f"w/cache{i}")

asyncio.run(main())
