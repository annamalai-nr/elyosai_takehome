2Premise and method

The take-home explicitly tells you the mock APIs are "intentionally imperfect... like real production APIs, they have undocumented behaviors, edge cases, and occasional failures." It also explicitly says /research "simulates real-world scenarios like database queries or AI processing." The hypothesis driving this research is that the engineers who built the mocks encoded the quirks they have personally been bitten by. So if we catalogue the documented public-domain quirks of the real APIs that conventionally occupy this slot, we have a tested checklist of likely planted behaviours.

Method: focused web research through GitHub issue trackers, vendor docs, vendor status pages, developer community threads, and production guides. Findings are cited inline by category in §16.

Three layers of quirks to be aware of:

  • Layer A — the mock endpoints themselves. Inspired by weather + search APIs in the wild. Covered in §4 and §5.
  • Layer B — the LLM loop in your own client code. Streaming tool calls, schema oddities, rate limits. Covered in §6 and §7.
  • Layer C — generic API messiness that crosses all categories. Covered in §8.

Strong submissions will demonstrate awareness of all three layers, but the take-home weighs Layer A heavily (it's the "API Discovery" evaluation criterion). Layer B is your code-quality evaluation. Layer C is what shows seniority.

4Weather APIs in the wild — documented quirks

These are the gotchas an engineer building an agentic-AI weather tool with real APIs has hit. They are the most likely template for the planted /weather quirks.

4.1 OpenWeatherMap — One Call 2.5 → 3.0 deprecation chaos

weather The single most-reported pain point in the weather-API ecosystem. OpenWeatherMap closed One Call API 2.5 in mid-2024. New API keys issued after the deprecation return 401 errors against the 2.5 endpoint — but client code with hard-coded v2.5 paths continues to send those requests for years. Surface bug: "my key works but I'm getting 401". Real cause: the URL is deprecated.

Sources: openhab-addons Issue #13314 (repeated 401s with new key due to deprecated URL); home-assistant/core Issue #119337 (Current Weather API broken/removed); pyowm Issue #404; sopel-weather Issue #52.

4.2 OpenWeatherMap — dramatic rate-limit shift

weather v2.5 gave 60 calls/minute free. v3.0 gives 1,000 calls/day total — that's ~0.69 calls/minute averaged. Apps that worked fine under v2.5 immediately hit rate limits under v3.0 even though they're at the same volume.

Equally important: the same account can have different subscription tiers per endpoint. Current Weather API and One Call API are separate subscriptions. Same key, different rate limits, different docs.

4.3 OpenWeatherMap — billing required for free tier

weather The v3.0 free quota (1,000/day) requires you to enter billing details to enable the endpoint, even though you stay under the free quota. Easy to miss in tutorials; surfaces as "endpoint returns 401 with a working key."

4.4 City resolution inconsistencies

weather Across weather APIs, ambiguous city names ("Springfield", "London" — there are multiple) resolve to different cities depending on input format, geolocation hints, and API version. Returned coordinates and timezone may not match the city the user meant. Best practice is to require explicit lat/lon, but agentic AI tools rarely do that.

4.5 Schema field omission vs null

weather Visual Crossing's documentation explicitly notes that fields are sometimes omitted entirely rather than returned as null. Defensive code that checks key existence works; code that assumes the key always exists crashes. This is generic across REST APIs but particularly notable in weather where some metrics (UV index, gusts) genuinely don't exist at all locations.

4.6 Format drift across endpoints

weather Same weather provider, different endpoints often return temperatures in different units (Kelvin vs Celsius vs Fahrenheit) depending on the units param default, the endpoint version, and the historical API surface. Apps that work for current weather but break on forecast often hit this.

4.7 Stale / cached responses

weather Weather APIs aggressively cache for cost reasons. Calling the same location twice within a few minutes returns identical data — sometimes identical timestamps even when nominally "current". Tests that rely on "the response changed" between calls fail.

4.8 5xx during partner-data hiccups

weather Weather APIs aggregate from underlying data partners (NOAA, ECMWF, etc.). When one partner is degraded, the API may return 500 / 502 for specific locations even when others work fine. PVOutput forum threads have repeated reports of "OpenWeatherMap problems" tied to upstream-source issues.

4.9 Free tier "soft" limits below documented numbers

weather Multiple weather providers (and AI APIs — see §6) throttle harder than their published limits. Documented 1,000/day might in practice fail at 700–800 if your traffic clusters in a short window.

5Web search / research APIs in the wild — documented quirks

These are the APIs that conventionally back the agentic-AI "research_topic" tool. Tavily is the most-used; Serper, Brave, Exa, and Perplexity Sonar are the major alternatives. The 3–8s latency profile of /research matches the real-world latency of these APIs.

5.1 Tavily — latency variance from p50 ~180ms to multi-second tail

search Tavily publishes p50 of ~180ms on basic search and ~998ms on advanced. In practice, advanced searches with include_raw_content=true and include_answer=true regularly run 2–5 seconds; cold cache or complex queries can push to 8+ seconds. Highly variable, which is exactly what the take-home's "3–8 seconds" simulation pattern looks like.

5.2 Tavily — credit-based pricing kills naive retries

search Basic search costs 1 credit; advanced search costs 2 credits. Free tier is 1,000 credits/month. Retrying a failed advanced search burns 2 credits regardless of whether the response was usable. Naive retry loops can drain a free tier in minutes.

5.3 Tavily — JS-rendered pages return flaky / empty content

search Tavily renders pages offline and caches results. JS-heavy sites (React/Vue SPAs that load content client-side) often return empty or partial content. The API "succeeds" — the body just has nothing useful in it. Surfaces as low-quality results, not as errors.

5.4 Serper / SerpAPI — Google scraping is structurally fragile

search Both providers scrape Google search results. When Google updates SERP layout/HTML, results temporarily go wrong. SerpAPI maintains a public roadmap of known bugs. Recent legal risk: Google v. SerpAPI lawsuit filed Dec 2025 creates regulatory uncertainty for the whole category.

5.5 Exa — strong on semantic similarity, weak on recency

search Exa's neural index excels at semantic similarity but scores low on freshness. Queries about recent events return stale results. Bench comparisons consistently flag this as Exa's main weakness.

5.6 Perplexity Sonar — hallucination in synthesis

search Perplexity's API does LLM-synthesised answers on top of search. The synthesised answer can hallucinate even when the underlying sources are accurate. Error messages on malformed requests are opaque (generic 400 without explanation), making debugging hard.

5.7 Common: snippet truncation

search All search APIs return snippets of bounded length. The same query can return different snippet lengths from the same source URL depending on internal logic. Code that depends on "this snippet will always contain X" breaks.

5.8 Common: rate-limit shapes differ per provider

search Each provider has its own 429 conventions. Tavily exposes credits remaining in headers; Serper uses a different header; Brave Search has its own. Generic 429 handlers fail across providers.

5.9 Common: dependency on third-party sources

search All real research APIs ultimately call out to web sources. When a target domain blocks scrapers, returns 403 to the API's user-agent, or rate-limits the API itself, results silently degrade. The API often returns a successful 200 with a thinner result set rather than reporting failure.

5.10 Authentication header conventions vary

search Tavily uses an api_key body parameter. Serper uses X-API-KEY header. Brave uses X-Subscription-Token. Perplexity uses Authorization: Bearer. The take-home uses X-API-Key — same as Brave Search's convention.

6Mainstream text-generation APIs — your LLM client layer

These quirks live in your client code, not in the Elyos mocks. They affect Layer B (the LLM loop) and are what the take-home's "implementation quality" criterion grades. Direct SDK use is what's expected; LangChain/LangGraph are explicitly discouraged.

6.1 OpenAI — streaming tool-call arguments arrive as partial JSON

llm When streaming tool calls, the function arguments field is delivered character-by-character as deltas. Most intermediate states are invalid JSON. You see {"qu, then {"query": "sal, then {"query": "sales rep. Clients must accumulate deltas and only parse when the tool-call block ends. Naive json.loads() on each chunk crashes.

Variant gotcha: some models/providers send the complete JSON in a single chunk, breaking parsers that expect incremental accumulation. The reverse also happens.

6.2 OpenAI — JSON Mode shape inconsistency

llm JSON mode guarantees parseability, not schema. Keys can go missing, appear at the wrong nesting level, or be spelled differently between requests. Use response_format=json_schema (Structured Outputs) for real schema enforcement.

6.3 OpenAI — 429 vs 5xx inconsistency on rate-limit conditions

llm Sometimes the same overload surfaces as a clean 429 with Retry-After; sometimes as 500/502 with no header. Rate limits are per-organization, not per-key. Unsuccessful requests still count toward the per-minute window. Naive retry compounds the problem.

6.4 OpenAI — 200 OK with body timeout (UND_ERR_BODY_TIMEOUT)

llm The /v1/responses endpoint can return 200 OK with headers arriving promptly, then the body hangs. Clients abort with a body-read timeout. Set explicit body-read timeouts.

6.5 OpenAI — SSE empty events crash naive parsers

llm Some SSE retry directives and comment lines come through as empty data: events. Clients that don't skip empties throw "unexpected end of JSON input."

6.6 OpenAI — mid-stream error events lack choices

llm When a stream errors mid-flight, the error object is shaped like {"error": {"message": "...", "code": 500}} — no choices array. Clients that expect every chunk to have choices crash on deserialization.

6.7 OpenAI — finish_reason: "stop" on truncated output

llm Truncation should return finish_reason: "length", but multiple reports show "stop" being returned on actually-truncated responses. Don't trust finish_reason alone — also compare actual length to expected.

6.8 Anthropic — 529 overloaded_error vs 429 rate limit

llm Two distinct error codes with similar shape but different semantics. 429 = your account hit a rate limit, slow down. 529 = provider overload, wait ~30s and retry. Treating them identically wastes rate budget or kills recoverable sessions.

6.9 Anthropic — empty text content blocks → 400

llm Submitting a message with empty text content blocks returns "messages: text content blocks must be non-empty". Common when an empty assistant turn slips into history.

6.10 Anthropic — unknown SSE event types break naive clients

llm Anthropic adds new event types over time. Per docs, consumers should ignore unknown types for forward compatibility. Many clients crash instead.

6.11 Anthropic — long max_tokens without streaming drops idle connection

llm Without streaming, intermediate networks drop idle connections during the model's "thinking" phase. Anthropic explicitly recommends streaming for requests >10 minutes.

6.12 Gemini — function-call argument streaming only on 3+ Pro

llm Gemini introduced streamFunctionCallArguments only in 3.x. Gemini 3.1 Flash-Lite does not support it. Code that assumes streaming arguments across all models breaks on Flash-Lite.

6.13 Gemini — safety filter behaves differently in stream vs non-stream

llm Documented bug in googleapis/python-genai Issue #929: the same prompt returns finish_reason = SAFETY in generate_content but finish_reason = STOP in generate_content_stream. Safety filtering is applied incrementally in streaming mode, with different evaluation logic.

6.14 Groq — three-dimensional rate limiting

llm Groq rate-limits across three axes: RPM (requests/min), TPM (tokens/min), and RPD (requests/day). The daily cap is what kills production usage on free tier (e.g., 1,000 RPD), not the per-minute limit. Per-model differences too (Llama 4 Maverick gets half quota; Gemma 2 9B gets 2.5× TPM).

6.15 Groq — streaming format quirks cause "tool tokens leaking into text"

llm Some integrations disable streaming for Groq because the format causes weird tool tokens to leak into text output. Pattern: treat streaming as optional rather than mandatory if compatibility matters.

6.16 Mistral / Cohere — tier limitations

llm Mistral's Experiment tier is prototype-only by terms of service. Cohere's free tier is non-commercial only. Code that works in dev may violate ToS in production.

6.17 Per-provider 429 signatures differ

llm Groq sends 429 with x-ratelimit-remaining-requests. Gemini sends 429 with retry-after. OpenAI sends 429 with various headers depending on endpoint. Generic 429 handlers fail across providers. The dispatcher needs per-provider parsing.

6.18 SSE idle disconnect on long tool-call loops

llm Long agentic loops where the LLM is "thinking" without emitting tokens for >30s get killed by intermediate infrastructure (load balancers, proxies, ngrok). Affects all providers.

7Routing / proxy layer — OpenRouter, LiteLLM

The take-home explicitly says "OpenAI, Anthropic, or similar." Many engineers reach for OpenRouter or LiteLLM to keep provider choice flexible. Those abstraction layers introduce their own quirks — worth knowing because Panos may ask about provider-neutrality at the on-site.

7.1 OpenRouter — silent routing to "bad" upstream servers

routing Documented in LiteLLM Issue #7887: OpenRouter sometimes routes requests to under-performing or broken upstream providers, surfacing as malformed responses, latency spikes, or quiet errors. You think you called Anthropic; OpenRouter actually routed to a fallback that's failing.

7.2 OpenRouter — tool_calls format inconsistency

routing A recent OpenRouter / Anthropic change requires tool_calls to be an array; some LiteLLM versions still send an object. Surfaces as 400 Bad Request: "expected array, received object". Cross-version compatibility matrix matters.

7.3 LiteLLM — silent 429 swallowing

routing LiteLLM Issue #8448: LiteLLM is not handling 429s from OpenRouter, surfacing as completion.choices[0].message.content = None with an error object you have to inspect separately. Clients that only check content think the call succeeded with empty output.

7.4 LiteLLM — internal default kwargs break OpenAI-compatible APIs

routing BaseOpenAI passes internal defaults like frequency_penalty to every request even when not specified. OpenAI-compatible providers that don't support all params reject the request with cryptic 400s. Hits especially on Anthropic-via-OpenRouter flows.

7.5 LiteLLM — endpoint mapping holes for custom model names

routing Custom model names like google/gemini-embedding-001 through OpenRouter return "unmapped LLM provider" errors. openai/whisper-1 via OpenRouter fails with provider mapping errors. The promise of "any model, one API" has edges.

7.6 Provider parameter passthrough is incomplete

routing OpenRouter exposes a provider argument to control upstream selection; LiteLLM doesn't pass it through cleanly in some versions. So even if you want to pin to one provider, you can't always.

7.7 Diagnosing failures: provider vs gateway

routing When a call fails, you have to figure out: did the gateway (LiteLLM / OpenRouter) reject it, or did the upstream provider? Error messages often don't distinguish cleanly. LiteLLM publishes an error-diagnosis guide; OpenRouter has its own error reference.

Engineering implication for the take-home: if you use LiteLLM in your CLI, decide deliberately whether to. The take-home says "use the provider's official SDK" — that's a hint to skip the router layer for this exercise. Bring it up as a trade-off you considered.

8Cross-cutting patterns — universal API gotchas

These show up across every API category and are likely encoded in the mocks at some level.

8.1 Idempotency on retries

cross-cut Only retry tools that are idempotent or read-only. Non-idempotent calls need an idempotency key. Both /weather and /research appear idempotent, so retries should be safe, but check.

8.2 Bounded exponential backoff + jitter

cross-cut Exponential (1s, 2s, 4s, 8s, 16s) + random jitter ±1s. Without jitter, distributed clients synchronise and cause thundering herds. Bound total retries.

8.3 Honour Retry-After when present

cross-cut Naive backoff that ignores the header retries too early and gets re-rate-limited.

8.4 Circuit breakers

cross-cut Track recent failure rate per endpoint. Open the circuit on high failure rate; allow occasional test calls during half-open. Important for the 100-concurrent-calls system-design discussion.

8.5 200 OK with error body

cross-cut Some APIs return 200 with an error field in the response body. Always inspect body shape before assuming success.

8.6 Schema drift / forward-compat tolerance

cross-cut Tolerate unknown fields; don't switch-case-exhaustively over event types or enum values.

8.7 Timeout that hangs vs errors

cross-cut Some APIs return a clean error on timeout; others just stop sending bytes. Always set explicit client-side read timeouts shorter than your end-user deadline.

8.8 Token / output budgets

cross-cut Cap input and output lengths. Don't trust finish_reason alone.

8.9 Stale / aggressive caching

cross-cut Identical queries may return identical responses for some window. Useful for cost, or wrong if you expected fresh data. Cache-buster params help diagnose.

8.10 Concurrent-request interference

cross-cut Some APIs serialise requests under the hood per-client; some have shared per-tenant resources that race. Probe under parallel load.

8.11 Soft limits below documented numbers

cross-cut Provider-side throttling is often harsher than published. The soft limit is the one that protects you.

8.12 Header conventions vary

cross-cut X-API-Key, Authorization: Bearer, X-Subscription-Token, api_key in body — all common. Note the take-home uses X-API-Key, matching Brave Search's convention.

9.3 Layer B — quirks in your LLM client loop (your code)

Even if the mocks behave perfectly, your handling of the LLM streaming + tool-calling loop must explicitly account for these. They're graded under "implementation quality."

  • Tool-call arguments arrive as partial JSON — never json.loads() mid-stream.
  • finish_reason not reliably indicating truncation.
  • Mid-stream error events that lack choices.
  • Empty SSE events from comment lines / retry directives.
  • SSE idle disconnect on long-running tool calls.
  • 429 / 529 / 5xx with provider-specific header conventions.
  • Cancellation: free resources, preserve partial response, stop downstream calls cleanly.

10Day-1 hypothesis checklist (run before writing handling logic)

Use this as your test plan. Run each item against /weather and /research in a quick exploratory script before writing the main implementation. Log everything to DISCOVERIES.md as you go.

A. Input edge cases (Layer A)

  1. Empty string: ?location= and ?topic=
  2. Single character, very long (5,000 chars), Unicode (e.g. 京都, emoji), whitespace-only
  3. SQL-injection-y inputs ('; DROP TABLE) — should be safely echoed back, not crash
  4. Known-bad locations / topics ("Atlantis", "Mars", "Made-up Town") — does it return a default, error, or fabricated content?
  5. Known-good locations with weird casing/punctuation ("LONDON", "lon-don", "Lon don")
  6. Ambiguous cities ("Springfield" — many real ones; "London" — Ontario vs UK)

B. Response schema (Layer A)

  1. Same input twice, 10 seconds apart — same response? Same keys? Same content?
  2. Different inputs — are response keys/shape consistent across them?
  3. Inspect every field type; check for null, missing keys, unexpected extra keys
  4. Check if response body ever contains {"error": "..."} alongside a 200 status
  5. For /weather: are units consistent across queries?
  6. For /research: does the summary's structure (paragraphs, citations, length) vary?

C. Latency profile (Layer A)

  1. Time 20 calls to /weather — what's the distribution? Any >1s? Tail at p95/p99?
  2. Time 20 calls to /research — what's the distribution? Any >8s? Any <3s? Tail shape?
  3. Idle for 60s, then call — cold-start spike?
  4. Rapid-fire 50 calls — does latency degrade? Rate limit?

D. Error handling (Layer A)

  1. Burst 200 calls in 5 seconds — do you ever see 429? Does it carry a Retry-After header?
  2. Run for 5 minutes, sampling — do you see 500/502 transients? At what rate?
  3. When you do see an error, what's the body? JSON? HTML? Plain text? Empty?
  4. Missing API key — what's the response? Wrong key?
  5. Do error bodies share a consistent shape, or do they vary?

E. Connection behaviour (Layer A)

  1. Set a 1-second client timeout — what happens? Clean error or hang?
  2. Cancel a /research request mid-flight (close the socket) — does the server complete the work anyway?
  3. Open a connection, send no data, wait 30 seconds — does the server close it?
  4. Run two /research calls concurrently — independent? Or serialised?

F. Caching / idempotency (Layer A)

  1. Repeat the same /research?topic=X 5 times in 10 seconds — same body? Same latency? Different request IDs?
  2. Add a cache-buster (random query param) — does behaviour change?
  3. Repeat the same /weather?location=London 5 times — same payload? Same timestamps?

G. Content-quality probes for /research

  1. Ask for a deliberately impossible topic ("the colour of the smell of Tuesday") — does it fabricate, return empty, or error?
  2. Ask for the same topic with subtle phrasing differences ("solar energy" vs "solar energy trends" vs "trends in solar energy") — same content or different?
  3. Ask for a recent topic ("2026 elections") — does it hallucinate dates / use a stale model knowledge cutoff?

H. LLM-layer specific (Layer B — your code)

  1. Confirm your tool-call argument parser handles partial JSON without crashing
  2. Confirm cancellation during streaming preserves partial response and frees resources
  3. Confirm finish_reason handling on truncation
  4. Confirm mid-stream error events don't crash your client (look for missing choices)
  5. Confirm 429 / 529 backoff with jitter; honour Retry-After when present

11Concrete probes you can run in 20 minutes

Drop this into a scratch probe.py before writing the main app. Replace X_API_KEY with the provided key.

import asyncio, httpx, time, json
from collections import Counter

BASE = "https://elyos-interview-907656039105.europe-west2.run.app"
HEAD = {"X-API-Key": "X_API_KEY"}

async def probe_input(client, endpoint, param, value, label):
    """One probe — record status, latency, body shape."""
    t0 = time.monotonic()
    try:
        r = await client.get(f"{BASE}/{endpoint}", params={param: value},
                             headers=HEAD, timeout=15.0)
        dt = time.monotonic() - t0
        body_preview = r.text[:300]
        try:
            j = r.json()
            keys = sorted(j.keys()) if isinstance(j, dict) else type(j).__name__
        except Exception:
            keys = "NOT_JSON"
        ra = r.headers.get("retry-after", "-")
        print(f"[{label}] {r.status_code} {dt:.2f}s retry-after={ra} keys={keys}")
        print(f"    body: {body_preview!r}")
        return (r.status_code, dt, keys, body_preview)
    except Exception as e:
        dt = time.monotonic() - t0
        print(f"[{label}] EXCEPTION after {dt:.2f}s: {type(e).__name__}: {e}")
        return (None, dt, None, str(e))

async def main():
    async with httpx.AsyncClient() as client:
        # A. input edge cases
        cases = [
            ("weather", "location", "", "weather/empty"),

13Appendix: voice APIs (for the on-site, not the take-home)

Brief notes on Deepgram and ElevenLabs quirks — kept because the on-site explicitly adds real-time audio via Deepgram STT and ElevenLabs TTS. Not relevant to the take-home itself.

13.1 Deepgram — top quirks

  • voice NET-0001: connection closes if no audio for 10 seconds. Send KeepAlive every 3–5s.
  • voice KeepAlive must be a text frame. Binary frame fails silently.
  • voice KeepAlive alone is not enough — at least one audio frame must be sent.
  • voice Interim vs final results: is_final: false vs true. Acting on interim is a common bug.
  • voice VAD/endpointing fails in noisy environments — speech_final=true may not fire.
  • voice smart_format=true intentionally delays finalisation for phone numbers, dates, etc.
  • voice WebSocket disconnect recovery requires audio buffering — replay buffered audio on reconnect.

13.2 ElevenLabs — top quirks

  • voice chunk_length_schedule — model stalls if text trickle is below the threshold.
  • voice WebSocket can be slower than REST over high-latency networks (711ms vs 478ms TTFB in real tests).
  • voice eleven_monolingual_v1 / eleven_multilingual_v1 deprecated.
  • voice Flash ≡ Turbo functionally, but Flash is lower latency.
  • voice 429 has two different messages: "too_many_concurrent_requests" or "system_busy" — same status code, different cause.
  • voice Default voices drift over time as ElevenLabs updates the catalogue — pin specific voice IDs in production.

13.3 OpenAI Realtime API — top quirks

  • voice Twilio + Realtime STT timeout race during initial greeting — buffer audio until both connections are ready.
  • voice Barge-in: server-side response.cancel + client must clear its own audio buffer.
  • voice SIP transport 521s intermittently on first connect (~5%).
  • voice Transport choice matters: WebRTC for browsers, WebSocket for server-orchestrated tools, SIP for telephony.
