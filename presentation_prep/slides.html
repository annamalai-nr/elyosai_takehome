Trade-offs

Reactive Retry vs Proactive Pacing

Shared rate-limit bucket: ~5 calls / 30s across both endpoints.

Considered

Proactive Pacer

  • Track call timestamps
  • Space requests evenly across window
  • Stay under budget preemptively
Requires knowing exact window size and call budget — neither documented. If wrong, burns calls and gets throttled anyway.
Chosen

Bounded Concurrency + Reactive Retry

  • asyncio.Semaphore per endpoint
  • max_concurrent = 1 — serialise same-endpoint calls
  • On throttle: read retry_after_seconds, wait, retry
  • Server stays the source of truth
Does not prevent throttles — responds to them. Multi-city requests run sequentially.

Key insight: "I chose to be honest about what I don't know rather than pretend I do."

Trade-offs

Code vs Prompt Responsibility

Where each quirk is handled depends on whether detection is deterministic.

Code — Deterministic

Reliable signal in the response

  • Shape A/B — temperature_c vs conditions
  • Location mismatch — requested ≠ returned
  • Cached data — cached: true
  • Timeout — if not data (empty {})
  • Throttle — status: "throttled" in body
Parsers catch it → deterministic → never wrong
Prompt — Judgment

No programmatic signal

  • Fabrication — Atlantis returns valid-looking data
  • Same shape, same fields, no error flag
  • Detection requires world knowledge

Alternatives ruled out:
Geocoder lookup → adds latency + external dependency
Hardcoded blocklist → always incomplete

LLM uses general knowledge → less deterministic → but never blocks a valid request

Trade-off: Prompt-level handling may miscaveat an obscure but real place. But it keeps the harness focused on what it can detect reliably, and avoids blocking legitimate requests for unusual locations.
